Why the iOS app can switch audio tracks but the web can't
The iOS player uses AVKit / AVPlayer which talks directly to the OS media pipeline. It can demux an MKV and expose every audio stream as a selectable track. Chrome's HTML5 video element runs in a sandboxed renderer — it only gets the audio track the OS decoder hands it for MP4 (usually track 0), and the audioTracks API is intentionally unimplemented. No amount of JavaScript can work around this.
The real fix — what will actually solve your issues
1
Subtitles: fetch from Stremio subtitle addonsdoable now
Call /subtitles/{type}/{id}.json from your installed subtitle addons (OpenSubtitles etc.), get VTT/SRT URLs, let the user pick one, load it in the player. Works with your current player OR Vidstack.
2
Audio tracks: show them as separate stream optionsworkaround
Some Torrentio entries are explicitly labeled with audio language (e.g. "1080p · English" vs "1080p · Spanish"). Surface these in the Sources panel as audio-language switches — the user picks a different stream, same content.
3
Vidstack is still worth it — but for DX, not audio tracks. It gives you a built-in subtitle panel UI, the source logo slot, better keyboard shortcuts, and you stop maintaining the custom player code. The subtitle fetching (fix 1) would plug straight into it.
